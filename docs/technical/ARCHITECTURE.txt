## Docker Deployment

**Why Docker:** Barcode scanning requires `pyzbar` Python library which depends on `libzbar0` system library. Render's native Python environment cannot install system packages. Docker solves this.

```dockerfile
FROM python:3.11-slim
RUN apt-get update && apt-get install -y libzbar0 && rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["gunicorn", "wsgi:app", "--timeout", "300", "--bind", "0.0.0.0:10000"]
```

**Render Services:**
| Service | Type | Status | Purpose |
|---------|------|--------|---------|
| `collectioncalc-docker` | Docker | **ACTIVE** | Production backend with barcode scanning |
| `collectioncalc` | Python | SUSPENDED | Legacy (saved $7/mo) |

## Environment Variables

Set on the **`collectioncalc-docker`** Render service.

| Var | Purpose |
|-----|---------|
| `EBAY_VERIFICATION_TOKEN` | Verification token for eBay's Marketplace Account Deletion endpoint challenge. Must EXACTLY match the token entered in the eBay developer portal. Used in `GET /api/ebay/account-deletion` to compute `sha256(challengeCode + verificationToken + endpoint)`. Also read by `dependency_monitor.check_ebay_account_deletion()` to independently recompute and verify the live endpoint's challenge response. 32вҖ“80 chars, alphanumeric + `_-`. |

> вҡ пёҸ eBay account-deletion challenge: the endpoint string hashed alongside the
> token is the hard-coded constant `EBAY_ACCOUNT_DELETION_ENDPOINT` in
> `routes/ebay.py` (`https://collectioncalc-docker.onrender.com/api/ebay/account-deletion`,
> no trailing slash). The registered URL in the eBay portal, that constant, and
> the portal's verification token must all match or the challenge hash will be wrong.
> Note: eBay's URL was historically registered against the now-SUSPENDED
> `collectioncalc` (no `-docker`) host вҖ” it must be updated to the `-docker` host.

## Recent Updates

### v4.9.0 - LPQ Texture Fingerprinting (Sessions 54-55 - Feb 20, 2026)
- **Added LPQ (Local Phase Quantization) as primary cross-camera copy-matching metric**
- LPQ uses STFT phase вҖ” blur-invariant, works across different cameras/lighting/backgrounds
- Key insight: discriminative signal lives in BORDER WEAR (corners, spine ticks, edge chips), not paper fiber or halftone dots
- Border-only LPQ promoted to primary: 4.7x stronger separation than full-image LPQ
  - Border-only: SAME=0.123, DIFF=0.586-1.375 (gap +0.463)
  - Full-image:  SAME=0.055, DIFF=0.154-0.335 (gap +0.099)
- Thresholds: border chi2 < 0.25 вҶ’ SAME_COPY, > 0.40 вҶ’ DIFFERENT_COPY
- LPQ override logic: prevents false same_copy from spurious border inliers (background texture)
- New functions in slab_guard_cv.py: _compute_lpq_histogram(), _compute_lpq_distance()
- Integrated into both compare_covers() and compare_covers_with_vision()
- In marketplace mode: LPQ serves as tiebreaker when Vision is uncertain
- 7 texture approaches tested (bandpass, phase correlation, Gabor, LPQ, DCT, gradient, diff RMS) вҖ” only LPQ worked
- 15+ failed approaches documented as institutional knowledge in slab_guard_cv.py docstring

### v4.8.0 - SIFT Copy Matching & Marketplace Mode (Sessions 48-53 - Feb 18-20, 2026)
- **4-tier matching pipeline:** Composite hashing вҶ’ Edge strip hashing вҶ’ SIFT-aligned edge IoU вҶ’ Claude Vision
- SIFT-based image alignment with FLANN matching + RANSAC homography
- Edge IoU: strict (pixel-exact) + dilated (3px tolerance) on 4 border strips
- Border inlier counting: SIFT inliers in border regions discriminate same-copy (3-4) from diff (0)
- Claude Vision "difference finder": structured checklist prompt with zoomed corner/edge crops
- Marketplace mode: Vision as primary verdict (quant metrics unreliable cross-camera)
- Auto-rotation: EXIF transpose + aspect ratio heuristic for rotated phone photos
- Void region detection: black warp artifacts excluded from IoU and Vision crops
- New file: routes/slab_guard_cv.py (CV engine)
- New file: routes/fingerprint_utils.py (shared auto-orient, preprocess)

### v4.7.0 - Composite Fingerprinting Upgrade (Session 46 - Feb 17, 2026)
- **Upgraded Slab Guard from single pHash to multi-algorithm composite fingerprinting**
- Now uses pHash + dHash + aHash + wHash (4 algorithms, 256 bits per angle)
- Full multi-angle composite: 1024 bits total across 4 photo angles
- Tested: same comic re-photo = 99/1024 (MATCH), different copy = 408/1024 (NOT MATCH)
- Marketplace scenario (front cover only): 83-bit separation вҖ” works with just 1 eBay photo
- registry.py: Generates composite fingerprints for all angles, stores as JSONB
- monitor.py: Composite matching with fallback to legacy pHash
- New thresholds: CRITICAL <70, PROBABLE <73, DISMISS >77 (per-angle composite)
- Database: Added fingerprint_composite JSONB column + GIN index (deployed)
- Backward compatible: existing pHash-only registrations still matched via legacy fallback
- Confidence score now scales with angle count (1 angle=70%, 4 angles=95%)
- New file: db_migrate_composite_fingerprints.py
- New file: COMPOSITE_FINGERPRINT_TEST_RESULTS.txt
