## ARCHITECTURE OVERVIEW

### Signature Identification System (v1 deployed / v2 pre-deploy)

**Current v1 (deployed, routes/signatures.py):**
- 4 endpoints: POST /match, POST /identify, GET /db-stats, GET /signed-sales
- /match: single Sonnet call, unknown sig vs ~46 reference images (2 per artist × 23)
- /identify: Haiku detect + Sonnet match against PostgreSQL candidates
- Local: signatures_db.json (23 artists, 97 images); Production: 43 artists, 172 images in R2
- Cost: ~$0.03/match call (gets expensive as artist count grows toward 100)

**Orchestrator v2 (pre-deploy, routes/signature_orchestrator.py):**
- Blueprint: /api/signatures/v2/match and /api/signatures/v2/match/stats
- Model: Claude Opus 4.6 (claude-opus-4-6)
- Architecture: PostgreSQL metadata pre-filter (100 → 15 candidates) → R2 image fetch → 3 parallel Opus calls (temp 0.2/0.5/0.7) → confidence aggregation → flags
- New DB table: signature_identification_log (review queue, confusion tracking)
- System prompt: prompts/signature_identification_system.md (loaded at runtime, inline fallback)
- Flags: low_confidence_match, high_confusion_pair, context_mismatch, stability_score per creator
- Known confusion pair: Jim Lee vs 3 other creators (being addressed by publisher_affiliations pre-filter)

**New DB columns needed (migration pre-deploy):**
- creator_signatures: career_start, career_end, publisher_affiliations (text[]), signature_style, reference_image_count, active, notes
- signature_images: r2_key, sort_order, source_url, approximate_year, image_notes
- Auto-trigger: trg_update_image_count keeps reference_image_count in sync
